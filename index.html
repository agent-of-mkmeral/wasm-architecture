Strands SDK: The WASM Shift

From N independent SDKs to 1 WASM core + N thin bindings

🏗️ The Problem

Multiple SDKs (Python, TypeScript) each implementing the full agent loop independently. Feature parity is hard. Bugs get fixed in one but not the other. Each SDK is 170+ files.

✨ The Solution

One core written in TypeScript, compiled to WASM via Component Model. Language-specific hosts provide thin bindings (~25 files). The WIT contract defines the boundary.

📦 What Ships

A .wasm binary artifact ships inside each language package (e.g., Python wheel). The host instantiates the WASM component and communicates via WIT imports/exports.

🔑 Key Insight

The boundary is data-only. Live objects (connections, callbacks, clients) cannot cross. Config and results (JSON/primitives) can. This splits responsibilities cleanly.

Model Provider Architecture

How model calls work through WASM and what changes for customization

Config Flow Through Boundary

// Python Host
OpenAIModel(
  model_id="gpt-4o",
  api_key="sk-...",
  client_args={"base_url": "https://my-endpoint.com/v1"}
)
        ↓  serializes to WIT record
        
// WIT Boundary (openai-config record)
{
  model_id: "gpt-4o",
  api_key: "sk-...",
  additional_config: '{"clientConfig":{"baseURL":"https://my-endpoint.com/v1"}}'
}
        ↓  deserialized in TS

// WASM Guest (TypeScript)
new OpenAI({
  apiKey: "sk-...",
  ...clientConfig  // ← baseURL lands here!
})

✅ Works with WASM

✅ client_args={"base_url": "..."}
✅ model_id + api_key config
✅ Environment variables (OPENAI_BASE_URL)
✅ timeout, maxRetries via clientConfig
✅ Any OpenAI-compatible endpoint via baseURL

❌ Breaks with WASM

❌ client=AsyncOpenAI(...) injection
❌ Subclassing OpenAIModel
❌ Custom Model base class (MLX, etc)
❌ Wrapper patterns (RetryBedrockModel)
❌ Custom HTTP interceptors/proxies

🏭 Who's Affected

NVIDIA NeMo Agent Toolkit

Client injection + subclass override for NIM endpoints. Both patterns break.

Tenstorrent

Uses client_args={"base_url": ...} — continues working perfectly.

strands-mlx (community)

Custom Model subclass for Apple Silicon. Needs TS port or host-side model.

AlloyModel (community)

Multi-provider routing. Complex pattern incompatible with single WASM model.

Responsibility Split

Python Host Owns:

  • • User-defined tools (@tool, MCP)
  • • MCP/A2A client lifecycle
  • • Multiagent orchestration
  • • Hook receivers (Python callbacks)
  • • Structured output (Pydantic)
  • • Plugin system

WASM Core Owns:

  • • Agent event loop (stream, retry)
  • • Model HTTP calls (WASI)
  • • Vended tools (bash, editor)
  • • Conversation management
  • • Session persistence
  • • Hook emission (lifecycle bridge)

Migration Impact

What works, what breaks, and recommendations

✅ No Change Needed

  • • Standard model config (model_id + api_key)
  • • Custom endpoints via base_url config
  • • Environment variable auth
  • • @tool decorated functions
  • • MCP tool connections
  • • Basic hooks (lifecycle events)
  • • Agent() constructor with config

❌ Breaking Changes

  • • Client injection (client=AsyncOpenAI(...))
  • • Model subclassing (NIMCompatibleOpenAIModel)
  • • Custom Model implementations (MLXModel)
  • • ConversationManager.reduce_context()
  • • Direct session state access
  • • Wrapper patterns (RetryBedrockModel)
  • • Community providers (12→4 built-in)

🔮 Future Solutions

  • • model-provider WIT import — host-side models
  • • TS ports of community providers
  • • additional_config escape hatch expansion
  • • Plugin-based model middleware
  • • Content formatter hooks in WIT

💡 Key Takeaway

The WASM shift trades flexibility (subclassing, injection, monkey-patching) for consistency (fix once → all languages, automatic parity, single source of truth). The additional_config JSON passthrough covers most custom endpoint needs. For advanced patterns (custom model implementations, content formatters), a future model-provider WIT import would restore host-side extensibility without sacrificing the core benefits.

Feature Development Workflow

How a new TS feature reaches Python bindings through WASM

🔄 End-to-End Flow: New Feature → All Languages

Adding a feature (e.g., a new model provider, conversation strategy, or agent method) follows this pipeline:

1
Implement in TS SDK — Add feature in strands-ts/src/ (models, tools, agent logic)
strands-ts/
2
Update WIT contract — If feature needs new types/methods crossing the boundary, edit wit/agent.wit
wit/
3
Update bridge — Wire new WIT exports/imports in strands-wasm/entry.ts
strands-wasm/
4
Build WASM — Run cd strands-wasm && npm run build → produces dist/strands-agent.wasm
build
5
Regenerate Python types — Run cd strands-py && generate-types (componentize-py reads WIT)
strands-py/
6
Update Python host — Add Python API surface in strands-py/strands/ (config classes, Agent wrapper methods)
strands-py/
7
Copy WASM artifact — Place strands-agent.wasm into strands-py/strands/_wasm/
ship
8
Test & Release — Integration tests run Python host ↔ WASM guest end-to-end
CI

📋 Decision: Does it need WIT changes?

NO WIT change needed if:

  • • Internal optimization (faster streaming, better retry)
  • • Bug fix inside existing event loop
  • • New model provider (uses existing model-config variant + additional-config)
  • • New vended tool (executes entirely inside WASM)
  • • Conversation manager improvement (config already has generic fields)

YES WIT change needed if:

  • • New method on Agent resource (e.g., agent.pause())
  • • New stream event type (e.g., reasoning-delta)
  • • New config record fields (e.g., new model provider variant)
  • • New import from host (e.g., model-provider)
  • • New lifecycle event type

⚡ Fast Path (No WIT Change)

If the feature is entirely inside TS (e.g., better retry logic, new model support via existing config): 

# 1. Implement in strands-ts/
cd strands-ts && # edit src/...
npm run build

# 2. Rebuild WASM (no WIT change)
cd strands-wasm
npm run build  # → dist/strands-agent.wasm

# 3. Copy artifact to Python package
cp dist/strands-agent.wasm \
   ../strands-py/strands/_wasm/

# 4. Test
cd ../strands-py && pytest

Python bindings don't change — new behavior "just works" because the WASM binary is updated.

🔧 Full Path (WIT Change Required)

If the feature adds new boundary-crossing types or methods: 

# 1. Edit WIT contract
vim wit/agent.wit  # add types/methods

# 2. Regenerate TS bindings
cd strands-wasm
npm run generate  # jco guest-types

# 3. Implement in TS + update entry.ts
cd strands-ts && # implement feature
cd ../strands-wasm && # wire in entry.ts
npm run build

# 4. Regenerate Python types
cd ../strands-py
generate-types  # componentize-py bindings

# 5. Update Python host API
# - Add config classes in models/ or agent/
# - Update _wasm_host.py if new imports
# - Add user-facing API in agent.py

# 6. Copy WASM + test
cp ../strands-wasm/dist/strands-agent.wasm \
   strands/_wasm/
pytest

🧪 Concrete Example: Adding a New Stream Event

Goal: Add reasoning-delta event so thinking tokens stream to Python

Step 1 — WIT:

variant stream-event {
  text-delta(string),
  reasoning-delta(string),  // NEW
  tool-use(tool-use-event),
  ...
}

Step 2 — entry.ts:

if (ev.type === 'reasoningDelta') {
  return { tag: 'reasoning-delta', val: ev.text }
}

Step 3 — generate-types:

@dataclass
class StreamEvent_ReasoningDelta:
    value: str  # auto-generated!

Step 4 — Python host:

# _wasm_host.py - event dispatch
elif event.tag == "reasoning-delta":
    yield {"type": "reasoning_delta", 
           "text": event.value}

💡 Key Principles

WIT is the contract

All types/methods that cross the boundary MUST be defined in wit/agent.wit. This is the single source of truth that both TS and Python code generates from.

entry.ts is the adapter

The bridge layer maps between WIT's flat records/variants and the TS SDK's rich class hierarchy. It handles all the conversion logic.

Python types are auto-generated

generate-types runs componentize-py to produce Python dataclasses from WIT. You never hand-write boundary types — only the user-facing API layer.

WASM binary is the artifact

The .wasm file ships inside the Python wheel. Updating it (even without WIT changes) can deliver new behavior to Python users with zero Python code changes.

⚠️ Common Gotchas

  • • WIT is additive-only for compat — removing/renaming fields breaks existing hosts. Add new optional fields instead.
  • • additional-config escape hatch — Use JSON string passthrough for model-specific config without WIT schema changes.
  • • Regenerate BOTH sides — After WIT change: jco guest-types (TS) AND generate-types (Python). Miss one = type mismatch.
  • • WASM binary must match WIT — If Python's types.py expects a field the WASM doesn't send, you get runtime errors.
  • • Test with real WASM — Unit testing TS alone isn't enough. Integration tests must run Python ↔ WASM ↔ real model calls.
  • • Host imports are sync — tool-provider.call-tool() blocks the WASM guest. Keep host callbacks fast.
